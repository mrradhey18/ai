## ADDING A NEW CLIENT

**Time required: 5 minutes**

1. Duplicate `data/clients/dr-sharma-homeo/` folder
2. Rename to new slug (e.g. `city-eye-care`)
3. Edit `profile.json`:
   - `business.name`, `business.slug`
   - `business.city`, `business.area`
   - `business.googleReviewUrl` — get from Google Business Profile dashboard
   - `services` array — add/remove services with keywords
   - `seo.localKeywords` — city/area specific keywords
   - `language.probabilities` — adjust per patient demographic
4. Add to `admin/js/admin.js` → `knownClients` array:
   ```js
   { slug: 'city-eye-care', name: 'City Eye Care Clinic' }
   ```
5. Open admin panel → QR Generator → generate + print QR

That's it. No code changes. No redeployment needed (if already live).

## ENGINE ARCHITECTURE

### How a review is generated (full chain)

```
ReviewEngine.generate({ profile, serviceId, stars })
  │
  ├── LanguageEngine.resolve(profile)
  │     └── weightedPick({ hinglish:40, english:25, hindi:25, mixed:10 })
  │           → selects 'hinglish'
  │           → loads all 7 phrase files for hinglish (parallel fetch)
  │           → caches for session
  │
  ├── _pickTemplate(stars=5)
  │     → selects Template A: [intro, service, trust, emotional, outro]
  │
  ├── per slot: _buildPhrase(type, phrases, stars, language, values)
  │     ├── getPool(phrases.intros, 5) → 5star phrase array
  │     ├── pick fresh phrase (AntiSpam.isPhraseUsed check)
  │     ├── Utils.fillTemplate("...{{service}}...", { service: "Skin Treatment" })
  │     └── Humanizer.humanizeSentence(filled, 'hinglish', 5)
  │           → apply contractions / elongations / lowercase / typos
  │
  ├── Utils.joinSentences([...]) → full review text
  │
  ├── SeoInjector.inject(text, phrases.seo, profile, 5, service)
  │     → 55% chance: add location phrase ("{{city}} mein...")
  │     → 40% chance: add credibility phrase
  │     → collision check: skip if city already mentioned
  │
  ├── Humanizer.humanizeReview(text, phrases, language, stars, profile)
  │     → casual spacing, hindi period drop, Hinglish fillers
  │     → emoji placement (60% base + 20% boost for 5star)
  │
  ├── AntiSpam.validate(candidate)
  │     → similarity check vs session history (threshold: 0.65)
  │     → exact duplicate check
  │     → if failed: retry (max 5 attempts)
  │
  └── repeat × 5 → returns array of 5 unique reviews
```

## PHRASE BANK RULES

### {{placeholders}}
- `{{service}}` → replaced with selected service label
- `{{city}}`    → replaced with `profile.business.city`
- `{{area}}`    → replaced with `profile.business.area`

### Adding phrases
Open any `data/phrases/[language]/[type].json` → add to the correct star pool:
```json
"5star": [
  "Your new phrase here.",
  "Another phrase for 5-star reviews."
]
```

### Rules for good phrases
- Write exactly how a real patient would type
- No corporate language ("We strive to provide...")
- Include `{{service}}` naturally in service phrases
- SEO phrases: max 1 location mention, sounds natural
- Hinglish: Roman script only, natural mixing
- Hindi: Devanagari only, no Roman mixing
- Mixed: genuine code-switching at grammatical boundaries

## CONFIGURATION REFERENCE

### profile.json — language probabilities
```json
"language": {
  "probabilities": {
    "english":  25,
    "hinglish": 40,
    "hindi":    25,
    "mixed":    10
  }
}
```
Values must sum to 100.

### profile.json — emoji settings
```json
"emoji": {
  "enabled": true,
  "probability": 60
}
```
`probability`: 0–100. 0 = never, 100 = always.

### profile.json — rating behavior
```json
"rating": {
  "5": { "tone": "emotional", "emojiBoost": true,  "sentenceLengthBias": "longer"  },
  "4": { "tone": "positive",  "emojiBoost": false, "sentenceLengthBias": "medium"  },
  "3": { "tone": "neutral",   "emojiBoost": false, "sentenceLengthBias": "shorter" }
}
```

## SCALABILITY GUIDE

### Current capacity (JSON-based)
- **1–50 clients**: works perfectly as-is
- **50–100 clients**: works, admin `knownClients` array gets long — manageable
- **100+ clients**: start planning the database migration below

### Scaling to 100+ clients — what changes

**Problem:** 100+ profile JSONs = manual `knownClients` maintenance in admin.js

**Fix:** Add a `clients/index.json` registry file:
```json
[
  { "slug": "dr-sharma-homeo",       "name": "Dr. Sharma Homeopathic Clinic" },
  { "slug": "green-wellness-clinic", "name": "Green Wellness Clinic" },
  { "slug": "city-eye-care",         "name": "City Eye Care" }
]
```
Admin loads this dynamically. Adding a client = add one line to index.json. No JS edits.

## DATABASE MIGRATION PATH

### When to migrate: 100+ clients or multi-user agency team

### Option A — Firebase (easiest)

```
Firestore structure:
  /clients/{slug}           → profile data (replaces profile.json)
  /phrases/{lang}/{type}    → phrase arrays (replaces phrase JSONs)
  /sessions/{userId}        → anti-spam session data

Replace:
  Utils.loadJSON(path)  →  firebase.firestore().collection('clients').doc(slug).get()
  Utils.storeSave()     →  firebase.firestore().collection('sessions').doc(userId).set()
```

Everything else stays identical. The engines don't change — only the data layer.

### Option B — Supabase (NexaFlow stack)

```sql
-- clients table
CREATE TABLE clients (
  slug        TEXT PRIMARY KEY,
  name        TEXT,
  city        TEXT,
  area        TEXT,
  google_url  TEXT,
  profile     JSONB,  -- full profile.json stored as JSONB
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- phrases table
CREATE TABLE phrases (
  id       SERIAL PRIMARY KEY,
  language TEXT,  -- english | hinglish | hindi | mixed
  type     TEXT,  -- intros | trust | service | etc.
  star     TEXT,  -- 5star | 4star | 3star | location | credibility
  text     TEXT,
  active   BOOLEAN DEFAULT TRUE
);

-- sessions table (anti-spam persistence)
CREATE TABLE review_sessions (
  session_id  TEXT PRIMARY KEY,
  reviews     JSONB,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

Replace `Utils.loadJSON()` with Supabase REST calls — same pattern you use in NexaFlow CRM.

### Option C — MongoDB

```js
// db structure
```

## KNOWN LIMITATIONS (MVP)

1. **No true auth** — admin panel is security-by-obscurity (URL not linked)
   → Add login page before sharing admin URL with clients

2. **Phrase edits require file access** — non-technical staff can't edit phrases
   → Build phrase editor UI in admin panel (Phase 2 feature)

3. **Analytics are zero** — no tracking of copy/redirect rates
   → Add analytics.js module for basic event tracking

4. **localStorage anti-spam** — clears if patient clears browser data
   → Move to server-side session for stricter enforcement

5. **QR codes use external API** — requires internet to generate in admin
   → Add qrcode.js library for offline QR generation
